**2.4 共識達成的瞬間**

經過多輪迭代，六個 AI 在一個問題上達成共識：

「THOOTB Zone 存續的必要條件是什麼？」

答案：**S = min(Stock Margin, FX Margin) ≥ 0**

其中：

- Stock Margin = (R-Pool 實物) / (日消耗量) - 安全天數

- FX Margin = (FG-Pool 外匯) / (日 Import 成本) - 安全天數

- S = 兩者的最小值

當 S ≥ 0：Zone 可以存續 當 S \< 0：Zone 進入危機

**為什麼是「最小值」？** 木桶原理：

- 即使你有很多外匯，如果實物不夠，Zone 也會崩潰

- 即使你有很多實物，如果外匯不夠買 Import，Zone 也會崩潰

- 最短的那塊板，決定了水位

**3.2 D-Pool：每日流通池**

**定義：** 每個公民每日可使用的消費額度

**核心規則：**

| **規則**          | **說明**                               |
|-------------------|----------------------------------------|
| 每人每日 TH 1,000 | （Simulator 預設值，實際 Zone 可調整） |
| 日終歸零          | 23:59 UTC 清零，未用完的消失           |
| 不可累積          | 今天的 TH 不能留到明天                 |
| 不可轉讓          | 不能把自己的 TH 給別人                 |

**Simulator 中的 D-Pool 計算：**

D-Pool 每日總量 = 人口 × TH 1,000

例：50 人 Zone

D-Pool 每日總量 = 50 × 1,000 = TH 50,000

每日結算：

├── 使用量（Activity）

├── 未使用量 → 歸零

└── Activity Rate = 使用量 / 總量

**健康指標：**

| **Activity Rate** | **狀態** | **意義**             |
|-------------------|----------|----------------------|
| 60-90%            | 健康     | 正常消費             |
| 30-60%            | 警告     | 消費偏低，可能有問題 |
| \< 30%            | 危機     | 嚴重異常             |
| \> 90%            | 警告     | 可能過度消費         |

**3.3 R-Pool：物資記錄池**

**定義：** Zone 內所有實物資源的追蹤系統

**核心規則：**

| **規則**  | **說明**                         |
|-----------|----------------------------------|
| 物資追蹤  | 每項物資有 ID、來源、狀態        |
| 循環驗證  | CHI ≥ 70%（地球）/ ≥ 80%（火星） |
| R&D + D&R | 強制循環認證                     |
| 不可隱藏  | 所有物資必須記錄                 |

**Simulator 中的 R-Pool 計算：**

R-Pool 狀態：

├── 總庫存量（Stock）

├── 日消耗量（Daily Consumption）

├── 日補充量（Daily Replenishment）

└── 庫存天數 = 總庫存 / 日消耗

關鍵指標：

Stock Margin = 庫存天數 - 安全天數（54 天）

**健康指標：**

| **庫存天數** | **狀態** | **意義**   |
|--------------|----------|------------|
| \> 90 天     | 健康     | 充足緩衝   |
| 54-90 天     | 警告     | 需要關注   |
| \< 54 天     | 危機     | 接近生死線 |

**3.4 FG-Pool：金融氣密艙**

**定義：** Zone 與外部世界交易的緩衝池

**核心規則：**

| **規則**    | **說明**                         |
|-------------|----------------------------------|
| 內外隔離    | TH 不直接對外，通過 FG-Pool 轉換 |
| 雙幣種      | 本地貨幣（TH）+ 外幣（FC）       |
| Export 收入 | 外幣進入 FG-Pool                 |
| Import 支出 | 外幣從 FG-Pool 支出              |

**Simulator 中的 FG-Pool 計算：**

FG-Pool 狀態：

├── 外幣餘額（FC Balance）

├── 日 Import 成本（Daily Import Cost）

├── 日 Export 收入（Daily Export Revenue）

└── 外匯天數 = 外幣餘額 / 日 Import 成本

關鍵指標：

FX Margin = 外匯天數 - 安全天數（54 天）

**健康指標：**

| **外匯天數** | **狀態** | **意義**   |
|--------------|----------|------------|
| \> 90 天     | 健康     | 充足緩衝   |
| 54-90 天     | 警告     | 需要關注   |
| \< 54 天     | 危機     | 接近生死線 |

**3.5 G-Pool：戰略儲備池**

**定義：** Zone 的戰略儲備，用於災難和長期投資

**核心規則：**

| **規則** | **說明**         |

E_g = Export Goods（貨物出口，賺外幣）

I = Import（進口，花外幣）

**白話：**

FX Margin = 賺的外幣 - 花的外幣

\> 0 → 外幣在增加（FG-Pool 成長）

= 0 → 外幣穩定

\< 0 → 外幣在減少（吃老本）

**Chapter 14: What Simulator Cannot Do**

**Simulator 不能做什麼**

**14.1 誠實的開場白**

Simulator 是一個工具。 一個很有用的工具。 但它不是水晶球。

這一章，我們要誠實地談談：Simulator 不能做什麼。

**14.2 六個根本局限**

┌──────────────────────────────┐

│ Simulator 的六個根本局限 │

├──────────────────────────────┤

│ 1. 不能預測人類行為的非理性 │

│ 2. 不能模擬真正的黑天鵝 │

│ 3. 不能捕捉社會動態的複雜性 │

│ 4. 不能替代人類判斷 │

│ 5. 不能保證數據的準確性 │

│ 6. 不能預見技術突破 │

└──────────────────────────────┘

**14.3 局限 1：人類行為的非理性**

**Simulator 假設：** 人們會按照經濟理性行動。

**現實是：**

- 恐慌囤貨 → Consumption 突然暴增 10 倍

- 謠言傳播 → 大規模退出潮

- 從眾效應 → 行為連鎖反應

- 情緒驅動 → 完全不按牌理出牌

**例子：**

Day 1：一切正常，S \> 0

Day 2：有人造謠「Zone 要倒了」

Day 3：30% 的人恐慌退出

Day 4：Export 能力崩潰

Day 5：S \< 0，真的要倒了

謠言變成自我實現的預言。Simulator 無法模擬這種「信心崩潰」。

**14.4 局限 2：真正的黑天鵝**

**Simulator 的「黑天鵝模式」模擬的是：**

- 已知類型的極端事件

- 歷史上發生過的衝擊

- 可以想像的最壞情況

**真正的黑天鵝是：**

- 你連想都想不到的事情

- 超出所有歷史經驗

- 定義上不可預測

2020 年之前，有多少經濟模型預測了「全球封城」？答案：零。

「如果你能模擬它，它就不是真正的黑天鵝。」

**14.5 局限 3：社會動態的複雜性**

**Simulator 簡化了：**

| **現實**     | **Simulator 假設**  |
|--------------|---------------------|
| 人有不同性格 | 所有 agent 行為相似 |
| 關係網絡複雜 | 簡化的交易關係      |
| 文化影響行為 | 忽略文化因素        |
| 權力動態存在 | 假設平等參與        |
| 信任需要建立 | 假設系統被信任      |

**例子：**

兩個人口相同的 Zone：

- Zone A：高度信任社會，鄰里互助

- Zone B：低信任社會，人人自保

Simulator 可能給出相同的 S 值。但 Zone A 在危機中會表現得好得多。

社會資本無法被量化。

**14.6 局限 4：不能替代人類判斷**

**Simulator 可以：**

- 計算 S 值

- 預測崩潰天數

- 顯示哪個指標異常

**Simulator 不能：**

- 決定該怎麼辦

- 權衡價值取捨

- 處理道德困境

**例子：**

Simulator 顯示：

「如果把 Import 從物資 A 換成物資 B，

FX Margin 會提高 20%」

但 Simulator 不會告訴你：

\- 物資 A 是藥品，物資 B 是奢侈品

\- 換掉 A 意味著有人會生病

\- 這是一個道德決定，不是數學問題

**Simulator 提供數據，人類做決定。**

**14.7 局限 5：數據準確性**

**垃圾進，垃圾出。（Garbage In, Garbage Out）**

如果輸入的數據錯了：

- Export 數字被高估

- Consumption 被低估

- Recycling 率是猜的

- 人口數字過時了

那麼 Simulator 的輸出就是錯的。不管方程式多漂亮。

**Simulator 的準確性 ≤ 輸入數據的準確性**

**14.8 局限 6：技術突破**

Simulator 基於現有技術水平。

如果明天發明了：

- 100% 效率的回收技術 → CHI 的計算要改

- 無限能源 → 整個資源約束要重新思考

- 完美的 AI 治理 → GSR 的意義改變

- ??? → 我們還不知道會改變什麼

Simulator 無法預見自己會被什麼淘汰。

S = +5,000 但連續下降 7 天

比

S = +1,000 但穩定上升

更值得擔心

- 應急預案

- 備用資源

- 快速反應機制

- 心理準備

**原則 5：定期校準**

每月檢查：

- 預測 vs 實際的差距

- 哪些假設需要更新

- 有沒有新的風險因素

- 參數是否需要調整

**calculateSustainability 函數**

const calculateSustainability = (params) =\> {

const {

consumption = 0, // C: Daily consumption (R_Stock outflow)

exportGoods = 0, // E_g: Goods export (R_Stock outflow)

exportService = 0, // E_s: Service export (no R_Stock change)

importValue = 0, // I: Import (R_Stock inflow)

production = 0, // P: Production gain

recycling = 0, // R: Recycling (FREE R_Stock inflow)

rStock = 0, // Current R_Stock

fgFC = 0, // Current FG-Pool FC

} = params;

// Stock Margin: 物資平衡

const stockMargin = importValue + production + recycling - (consumption + exportGoods);

// FX Margin: 外匯平衡

const fxMargin = exportService + exportGoods - importValue;

// System Sustainability Margin: 最弱的那條腿

const sustainabilityMargin = Math.min(stockMargin, fxMargin);

// Is Sustainable?

const isSustainable = sustainabilityMargin \>= 0;

// Days to Collapse (with edge case handling per 6AI feedback)

let daysToStockCollapse = Infinity;

let daysTofxCollapse = Infinity;

if (stockMargin \< 0) {

if (rStock \<= 0) {

daysToStockCollapse = 0; // Immediate collapse - no reserves

} else {

daysToStockCollapse = rStock / Math.abs(stockMargin);

}

}

if (fxMargin \< 0) {

if (fgFC \<= 0) {

daysTofxCollapse = 0; // Immediate collapse - no FX reserves

} else {

daysTofxCollapse = fgFC / Math.abs(fxMargin);

} }

const daysToCollapse = Math.min(daysToStockCollapse, daysTofxCollapse);

// Weakest Link

const weakestLink = stockMargin \<= fxMargin ? 'Stock' : 'FX';

// Resource Change Rate (Ṙ)

const rDot = stockMargin; // dR/dt

const rDotStatus = rDot \> 0 ? 'Growing' : rDot \< 0 ? 'Depleting' : 'Stable';

return {

stockMargin,

fxMargin,

sustainabilityMargin,

isSustainable,

daysToCollapse: daysToCollapse === Infinity ? '∞' : Math.round(daysToCollapse),

daysToStockCollapse: daysToStockCollapse === Infinity ? '∞' : Math.round(daysToStockCollapse),

daysTofxCollapse: daysTofxCollapse === Infinity ? '∞' : Math.round(daysTofxCollapse),

weakestLink,

rDot,

rDotStatus,

};};
